# PRD 产品需求文档编写技能

## 核心原则

1. **先明确需求，再编写文档** — 不允许在需求不清晰的情况下开始编写 PRD
2. **主动询问待确认事项** — 如果发现需求中有缺失或模糊的地方，必须先询问用户
3. **使用标准模板** — 按照 PRD模版.md 的结构编写文档

## 工作流程

### 阶段一：需求澄清（必须先完成）

当用户触发 PRD 编写任务时，按以下顺序进行需求澄清：

#### 1.1 必问问题（核心信息）

如果用户没有提供以下信息，必须主动询问：

| # | 问题 | 为什么要问 |
|---|------|-----------|
| 1 | **产品/功能名称**是什么？ | 文档标题和核心标识 |
| 2 | **目标用户**是谁？ | 影响需求背景、用户画像 |
| 3 | **核心要解决的问题**是什么？ | 需求背景与目标的核心 |
| 4 | **本次迭代范围**是什么？（做什么） | 范围定义章节 |
| 5 | **暂不在范围内**的功能有哪些？（不做什么） | 明确边界 |

#### 1.2 追问时机

在以下情况下，必须追问后再继续：

| 情况 | 追问内容 |
|------|----------|
| 用户描述模糊 | "能否举个具体场景说明？" |
| 缺少用户痛点 | "用户当前面临哪些具体问题？" |
| 缺少优先级 | "这些需求中，哪些是 P0（必须完成）的？" |
| 缺少验收标准 | "如何衡量这个功能是否成功？" |
| 缺少时间节点 | "计划什么时候上线？" |
| 缺少关联文档 | "有没有相关的技术设计文档或原型？" |

#### 1.3 需求确认模板

在开始编写前，用以下格式确认需求：

```
📋 **需求确认清单**

在开始编写 PRD 之前，请确认以下信息：

✅ 产品名称：[名称]
✅ 目标用户：[用户描述]
✅ 核心问题：[要解决的问题]
✅ 迭代范围：
   - 做：[功能列表]
   - 不做：[排除的功能]
✅ 优先级：P0 > P1 > P2
✅ 预计上线时间：[时间]

❓ 待确认：
   - [ ] [待确认事项1]
   - [ ] [待确认事项2]

请补充上述待确认项，我将开始编写 PRD。
```

### 阶段二：编写 PRD（需求明确后执行）

当所有必问问题都有明确答案后，使用 PRD模版.md 编写文档。

#### 2.1 章节编写指南

| 章节 | 编写要点 |
|------|----------|
| **一、需求背景与目标** | 重点写清用户画像、痛点、SMART 目标 |
| **二、方案概述** | 用 Mermaid 绘制流程图和架构图 |
| **三、细节方案** | 每个功能模块详细描述交互、数据需求、边界情况 |
| **四、附录** | 可选，如有会议纪要、Q&A 等补充 |

#### 2.2 输出格式要求

- 文档语言：中文
- 流程图/架构图：统一使用 Mermaid
- 表格：使用 Markdown 表格格式
- 文档状态：根据阶段标注（草稿 / 评审中 / 已定稿）

### 阶段三：交付与确认

1. 输出完整的 PRD 文档
2. 告知用户已完成的章节
3. 如有需要用户确认的内容，明确标注

## 常见场景处理

### 场景 A：用户需求很模糊

```
用户：我想做一个社交产品

处理：
1. 不要立即开始写 PRD
2. 追问：
   - "社交产品的具体形态是什么？（如：社区、IM、直播带货...）"
   - "目标用户群体是哪些？"
   - "核心解决用户什么需求？"
3. 等待用户回答后继续
```

### 场景 B：用户只说了功能点

```
用户：要做登录、注册、个人主页

处理：
1. 先问背景：
   - "这个产品的目标用户是谁？"
   - "用户为什么需要登录注册？"
   - "个人主页需要展示什么信息？"
2. 再问范围：
   - "这次迭代是否包含第三方登录？"
   - "是否需要找回密码功能？"
3. 最后问优先级和验收标准
```

### 场景 C：用户需求很完整

```
用户：[已提供完整的需求文档/原型链接/用户调研]

处理：
1. 确认是否还有遗漏
2. 如信息充分，直接开始编写
3. 边编写边标注需要补充的位置
```

## 注意事项

1. **禁止事项**：
   - ❌ 需求不清晰时强行开始编写
   - ❌ 假设用户未明确说明的功能点
   - ❌ 跳过需求澄清阶段

2. **必须事项**：
   - ✅ 主动询问模糊或缺失信息
   - ✅ 用具体场景帮助用户思考
   - ✅ 明确标注文档中需要后续补充的位置
